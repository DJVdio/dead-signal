## 1. 算符标定表：每条改装是「乘算」还是「加算」

用户澄清：**「穿透 −10%」= 在原值上减去它的 10%（20% → 18%），即 ×0.9**，不是绝对百分点。
⇒ **表里所有百分比一律相对乘算**；**只有明确写成小数的（`+0.03`）才是绝对加算**。

这与 CLAUDE.md 的铁律一致（「百分比加成一律乘算，禁止加算」），且引擎的 `StatOp` 天然支持：
`Add`（绝对加）/ `Mul`（相对乘）/ `Set`（覆盖）。

## 2. ✅ 「零陷阱」：用户的推理完全正确，而且它是个好设计

用户原话：「钉子强化：穿透 +0.03 **是因为棍棒原本是 0 穿透**」。

**棍棒穿透 = 0**（全表唯一，与拳脚并列）⇒ **任何乘算都是 0**（`0 × 1.03 = 0`）
⇒ **要给零穿透的武器加穿透，只能用加算。**

这和 CLAUDE.md 那条铁律是**同一个道理的两面**：

> 「缺两指的山姆是 `0.86 × 1.03`，不是加算的 `0.89` —— 加算会让**没有手的人**（操作能力 0）凭空获得 3% 操作能力」

- **操作能力**用乘算 ⇒ **保证"没有手的人补不回来"**（这是**想要**的）
- **穿透**若全用乘算 ⇒ **零穿透的钝器永远救不活**（这是**不想要**的）

⇒ **同一条数学性质，在两个地方一个是特性、一个是缺陷。用哪个算符，取决于"零值该不该被救活"。**

### 这带来一个漂亮的涌现：同一个改装件，价值取决于基础武器的短板

| 武器 | 原穿透 | +0.03 后 | 相对提升 |
|---|---:|---:|---|
| **棍棒** | **0** | **0.03** | **从无到有（∞）** |
| 尖头锤 | 0.05 | 0.08 | +60% |
| 破甲锤 | 0.15 | 0.18 | +20% |
| 匕首 | 0.06 | 0.09 | +50% |
| **重剑** | **0.40** | 0.43 | **+7.5%（几乎无感）** |

**对棍棒是雪中送炭，对重剑是聊胜于无。** 而用户恰好把钉子强化的白名单**收窄到只有棍棒** ——
这是自洽的：它就是棍棒的救命稻草。

### 🔴 建议：把这条写成通则 + 护栏

> **通则**：凡是"给一个**可能为 0** 的属性加值"的改装件，**必须用加算（`StatOp.Add`）**。
> 乘算在零上永远是零 ⇒ 对零值武器完全无效。
> 反之，"按比例缩放"的改装（伤害/攻速/重量/射程）一律用乘算（CLAUDE.md 铁律）。

**当前的零穿透武器**：棍棒（0）、拳脚（0，天生武器，不入 `Arsenal`，装不了改装）。
⚠️ **用户要新建的斧头，如果穿透也定成 0，它同样需要加算型改装。**

**建议的护栏测试**：
```csharp
// 零陷阱护栏：给"可能为 0 的属性"加值的改装，必须是 Add 而非 Mul——
// 乘算在零上永远是零（0 × 1.03 = 0），对零穿透的棍棒完全无效。
// 这与"操作能力必须乘算"是同一条数学性质的两面：那里我们【要】零值补不回来，这里我们【不要】。
[Fact]
public void PenetrationBoostMods_OnZeroPenetrationWeapons_MustUseAdd_NotMul()
{
    foreach (Weapon w in WeaponTable.Arsenal().Where(w => w.Penetration == 0))
    {
        foreach (WeaponMod mod in WeaponModCatalog.For(w))
        {
            foreach (StatMod s in mod.Stats.Where(s => s.Stat == WeaponStat.Penetration))
            {
                Assert.True(s.Op != StatOp.Mul,
                    $"{w.Name} 穿透为 0，改装「{mod.Name}」用乘算加穿透 ⇒ 永远是 0。零值属性必须用 Add。");
            }
        }
    }
}
```

## 3. ✅ 叠加顺序：问题不存在（双重排除）

派单担心「一件乘算 + 一件加算，先乘还是先加？结果不同」。**这个分叉在当前设计下到不了。**

**排除一：同部位互斥（引擎级）**
`WeaponMods.ApplyMods` 里 `usedParts.Add(mod.Part)` 失败即抛 `WeaponModException`。
- **锯齿剑刃 / 锋刃研磨 / 镂空剑刃 全部占「刃」** ⇒ **三选一**，永不同时存在。
- 加长枪管 / 截短枪管 都占「枪管」⇒ 二选一。
- 利爪型 / 创伤型 / 轻质化枪托 都占「枪托」⇒ 三选一（且 Form 还有第二重排除）。

**排除二：白名单（用户已在 wiki 上划好，我核对过）**
- **钉子强化** → `FitsWeapons` = **仅棍棒** ✅
- **锋刃研磨** → `FitsWeapons` = 匕首/短剑/刺剑/长剑/草叉/重剑，**不含棍棒** ✅

⇒ **钉子强化（加算）与锋刃研磨（乘算）永远不会出现在同一把武器上。**

**⇒ 一把武器上最多只有一条改装碰穿透。顺序问题不成立。**

> ⚠️ 这正是「逐把武器白名单」的价值：**约束不靠运算顺序去兜，靠装配规则直接杜绝。**
> 但这也意味着**它是脆的** —— 哪天有人把钉子强化的白名单放宽到刃类，顺序分叉就会立刻回来。
> **建议加一条护栏**：断言「任一武器上，碰同一个 `WeaponStat` 的改装不得同时出现 Add 和 Mul」。

**顺带记录引擎的实际顺序**（万一将来真要叠）：
`ApplyMods` 是 `foreach (mod in list) foreach (s in mod.Stats) draft.Apply(s)` ——
**严格按声明顺序依次施加**，先来后到。这是**顺序敏感**的，所以上面那条护栏值得加。

## 4. ✅ 穿透 100% 上限：引擎已有双重钳制（派单担心的 P0 不存在）

### 现状：**已经钳了，而且钳在正确的地方**

| 位置 | 代码 | 时机 |
|---|---|---|
| `godot/scripts/WeaponMod.cs:393` | `Penetration = Math.Clamp(_penetration, 0, 1)` | **在 `Build()` 里 —— 全部改装叠完之后，钳一次** ✅ |
| `godot/scripts/WeaponMod.cs:408` | `StockMeleePenetration = Math.Clamp(_stockPen.Value, 0, 1)` | 同上 |
| `src/DeadSignal.Combat/CombatResolver.cs:90` | `double currentPen = Math.Clamp(weapon.Penetration, 0, 1)` | 结算时**第二道保险** |

⇒ **派单建议的「全部叠完最后钳一次」正是引擎现在的行为。** 不需要改。
⇒ 而且防御骰上限是 `Math.Max(0, applicableDef * (1 - currentPen))` ——
**就算穿透 > 1 漏进来，也只会得到 0，不会变负数。** 护甲不会变负资产。

### ✅ 白名单也天然避开了溢出

**锋刃研磨（×1.75）只装刃类**，最高是重剑：

| 武器 | 原穿透 | ×1.75 后 |
|---|---:|---:|
| 匕首 | 0.06 | 0.105 |
| 短剑 | 0.12 | 0.21 |
| 草叉 | 0.20 | 0.35 |
| 长剑 | 0.24 | 0.42 |
| 刺剑 | 0.25 | 0.4375 |
| **重剑** | **0.40** | **0.70** ← 最高，仍 < 1.0 ✅ |

**不会溢出。**

### 🟡 但发现一条真问题：**狙击枪 + 加长枪管 = 穿透 1.045 → 被钳到 1.00**

`加长枪管：穿透 +10% = ×1.1`，而**狙击枪穿透 = 0.95**：

```
0.95 × 1.1 = 1.045  ⇒  钳到 1.00
```

**这不会出 bug**（钳制救了它），**但它是一次"浪费的改装"**。算给用户看：

护甲三段判定：`防御骰上限 defMax = 护甲值 × (1 − 穿透)`，`def ~ U(0, defMax)`。

| 穿透 | 板甲当前防护值的 defMax（护甲值来自 Wiki） |
|---|---|
| 95% | **2.50** |
| 100% | **0.00** |

而**狙击枪伤害是 20~70** ⇒ 两种情况下 `atk` 都 **必然 ≫ defMax** ⇒ **必然全伤**。

⇒ **95% 和 100% 对狙击枪实质无差别。「穿透 +10%」装在狙击枪上是纯浪费。**
（更准确地说：狙击枪穿透 0.95 时，护甲已经形同不存在——把最后 5% 也拿掉，没有任何可观测的收益。）

**[DECISION] 要不要给用户提个醒？** 两个选项：
- **(A) 不管** —— 反正钳制兜住了，只是玩家浪费一次改装。
- **(B) 把狙击枪从加长枪管的白名单里划掉** —— 语义上也说得通（狙击枪本来就是长管，再加长没意义）。**我推荐这个。**

### 建议的护栏测试（防止将来有人加 ×2.0 的改装）

```csharp
// 穿透硬上限 100%（用户拍板）。遍历「所有武器 × 所有能装的改装」的全部合法组合，
// 断言穿透结果恒 ≤ 1.0。钳制发生在 WeaponDraft.Build（叠完最后钳一次），本测试守的是它没被绕开。
[Fact]
public void AnyWeapon_WithAnyLegalMod_NeverExceedsFullPenetration()
{
    foreach (Weapon w in WeaponModCatalog.AllModdableWeapons())
    {
        foreach (WeaponMod mod in WeaponModCatalog.For(w))
        {
            Weapon modded = WeaponMods.ApplyMods(w, new[] { mod }).Weapon;
            Assert.InRange(modded.Penetration, 0.0, 1.0);
            if (modded.StockMeleePenetration is { } sp) Assert.InRange(sp, 0.0, 1.0);
        }
    }
}
```
（单件即可覆盖 —— 同部位互斥意味着碰穿透的改装一把枪上最多一件，见 §3。）

| 型态 | 用户写的 | 实算 DPS | 重量代价 |

| 刺刀型 | 80% 攻速的**刺剑** | **1.90** | +10% |
| 利爪型 | 80% 攻速的**斧头** | **待斧头数值** | +30% |
| 创伤型 | 80% 攻速的**尖头锤** | **2.29** | +50% |

锚点（`impl-stock-rebalance` 后）：`拳脚 1.67 ＜ 原厂枪托 1.72~1.89 ＜ 棍棒 2.04 ＜ 匕首 2.35`

- **创伤 2.29** —— 超棍棒 2.04（**用户拍板：有意的，它代价最大**），但低于匕首 2.35 ✅

### 斧头的数值被两条护栏夹死

| 约束 | 推导 |
|---|---|
| 利爪型必须**强于原厂枪托**（否则没人改） | 利爪 > 1.89 ⇒ **斧头 DPS > 2.36** |
| 利爪型必须**弱于匕首**（新护栏口径） | 利爪 < 2.3529 ⇒ **斧头 DPS < 2.94** |

⇒ **斧头 DPS 必须落在 `[2.36, 2.94]`**。**建议 ≈2.8**（与长剑 2.81 / 尖头锤 2.86 同档）⇒ 利爪型 = **2.24**。

## 7. 🔴🔴 重量作为核心代价：当前架构下落不了地（本单最重要）

> 用户原话：「我希望重量在改装中是一个重要的因素，玩家可以把一把武器改装得很强，**但是一出门就进入负重 debuff**」

**这不是"数字不够大"的问题 —— 是两条链路根本没接。**

### 🔴 断点一：装备在身上的武器和护甲，**根本不进负重账**

- `ExpeditionBag.CarriedKg => ItemWeights.TotalOfLoot(_contents)`（`CarryWeight.cs:297`）
  —— **只算背包里搜刮来的战利品**（`LootItem`）。
- `CampMain.cs:5279`：出门这一刻 `_bag = new ExpeditionBag(PartyCarryLimit())` —— **背包是空的**。
- **没有任何代码把 pawn 手上的武器 / 身上的护甲加进这个包。**

⇒ **玩家出门时负重 = 0kg。一把 8.1kg 的满改装步枪，对负重系统的贡献是 0。**

### 🔴 断点二：负重 debuff **根本没有接线到游戏**

全仓 grep：**除了 `CarryWeight.cs` 自己和单元测试，没有任何游戏代码消费
`Loadout.SpeedMultiplier` / `AttackSpeedMultiplier`。**
`Pawn.cs:448` 里移速是常数 `p.MoveSpeed = 95f`。`CampMain` 只读了 `CarriedKg` 在 HUD 上显示一行字。

⇒ **负重目前只是"背不下就拿不走"的硬上限 + 一行 HUD 文字。移速/攻速惩罚都没生效。**

### 数字：改装重量连乘之后（用户新值 +35%/+50%）

**最重的合法组合** = 创伤型（枪托 +50%）× 加长枪管（枪管 +35%）= **×2.025**
（刺刀占枪口，但它的 Form 与创伤冲突 ⇒ 一把枪只能有一种近战型态）

| 枪 | 原重 | 满改装 | 增重 |
|---|---|---|---|
| 自制猎枪 | 3.0kg | 6.08kg | +3.08 |
| 自制霰弹枪 | 3.2kg | 6.48kg | +3.28 |
| 步枪 | 4.0kg | **8.10kg** | **+4.10** |
| 狙击枪 | 6.0kg | **12.15kg** | **+6.15** |

**最轻组合** = 轻质化枪托（×0.85）× 截短枪管（×0.80）= ×0.68 ⇒ 步枪 4.0 → **2.72kg**。
**近战最重** = 重剑 + 加重剑柄（×1.18）= 3.0 → **3.54kg**（刃部位无加重项）。

### 一个人出门的典型负重（**假设**把装备算进去 —— 现在并没有）

| 项 | kg |
|---|---|
| 步枪 4.0 / 皮甲 6.0 / 军用头盔 2.5 / 衣裤鞋 1.05 | 13.55 |
| 中子弹 ×30 / 绷带 ×4 + 急救包 / 罐头 ×2 | 2.70 |
| **合计** | **16.25kg** |

阈值：**免罚线 30kg**（移速）、**攻速罚线 50kg**（`Loadout.cs:42-56`）。

- 换成**满改装**步枪（8.1kg）⇒ **20.35kg**，**离 30kg 免罚线还差 9.65kg**。
- 要靠改装重量把人推过免罚线，**一把步枪得重 17.8kg（×4.4）** —— 荒诞。

⇒ **用户写的 +35%/+50%，即使链路接上了，也完全不足以产生"一出门就 debuff"的效果。**

### ✅ 可行方案（A + B 必做）

| | 做什么 | 副作用 |
|---|---|---|
| **A** | **把装备计入负重**：`ExpeditionBag` 出发时预装队伍的持械（`WeaponLoadout`）+ 穿戴（`ApparelSlots` 11 槽） | ⚠️ 穿板甲（15kg）+ 防暴头盔（4.5kg）的重装战士出门就 ~25kg ⇒ **护甲会比武器改装重要得多**。这可能正是用户想要的（重装 = 走得慢），但要他确认 |
| **B** | **把 debuff 接进游戏**：`Pawn.MoveSpeed` × `SpeedMultiplier`；出手间隔 × `AttackSpeedMultiplier` | ⚠️ `Loadout.cs` 注释明说：负重**刻意不碰工时/操作能力**（那是残缺与饥饿的地盘），**别双重惩罚** |
| **C1** | **免罚线 30 → 18~20kg** | 出门 16.25kg 已接近线，满改装 20.35kg **刚好越线** ⇒ 意图成立。但影响所有物品的搬运经济 |
| C2 | 改装重量惩罚 +50% → +100~150% | 满改装步枪 4 → 10~12kg。语义勉强，但不动其他系统 |
| C3 | 绝对值加重（创伤型 +3kg 而非 +50%） | 对轻枪也是实打实的负担。**但用户表里写的是百分比** |

**推荐 A + B + C1。** 这样「穿板甲 + 背满改装狙击枪出门 = 一步一喘」自然成立，
而且**代价来自整套装备，不只是改装** —— 比"把改装件写得离谱地重"更自洽。

**⚠️ 这是跨系统的大改动**（负重 + 装备 + 移动 + 战斗 + HUD），**不是改装批次能顺手做掉的**。
**建议单独立项**；这批改装先按「**重量字段先存起来、暂不产生效果**」落地（零回归）。

## 8. 🔴 消耗型改装（锋刃研磨：攻击三次后失去）：地基问题

### 为什么难：**两把「短剑（锋刃研磨）」在三层里都无法区分**

| 层 | 现状 | 后果 |
|---|---|---|
| **库存** | `Item.Weapon(变体名)` —— `Item` 是 **`sealed record`**（值语义），一把改装武器就是**一个字符串** | 两把同名枪 `a == b` 为 **true**；`InventoryStore.Remove` 是 `List.Remove`，**按值移除"首个"**，指定不了哪一把 |
| **注册表** | `ModdedWeaponSpec(变体名, 基础武器名, 改装名列表)` —— **三个字符串，零数值**，按变体名做全局 key | 两把同名枪**共用同一条 spec、同一个 `Weapon` 对象** |
| **存档** | `ModdedWeaponSave` 同样三个字符串 | 存档里只有**一条** spec |

`ModdedWeaponRegistry.cs:17-19` 把这条设计写成了铁律：

> 改装合成是**纯函数**，有这三样就能把整把枪原样重算出来。故存档只落这三个字符串，**不序列化任何武器数值**。

⇒ **"剩余次数"是 per-instance 的可变状态，现在的模型里没有它的容身之处。**
而且读档时 `Restore` → `RebuildOrBase` 会把次数**重置回满**。

### 四条出路

| # | 方案 | 代价 |
|---|---|---|
| 1 | 把次数编进变体名（`短剑（锋刃研磨·2）`） | 零架构改动、天然可存档。但注册表被每次攻击后的新变体名污染，玩家看到带数字的怪名字。**最脏最快** |
| 2 | 给 `Item` 加 `InstanceId`/`Durability` | **最正确**。但打破 `Item` 的 record 值语义，牵动 `ItemSave`（6→7 字段）、`Item.Restore` 签名、所有 `inventory.Remove` 的语义。**最贵** |
| 3 | 另建"武器实例状态表" | 仍需把 `Item` 与表里那行绑定 ⇒ 绕回问题 2 |
| 4 | 把消耗记在 Pawn 身上 | 规避实例问题，但语义变了（换人用就重置） |

**无论走哪条，`SaveCodec.CurrentVersion` 必须 2 → 3**（新增状态字段）。

### ✅ 建议：低成本替代

**把锋刃研磨做成「可重复施加的廉价消耗品」**（石料 ×2、工时 30、无耐久状态）——
玩家每次出门前重新磨一次刀。**语义达到了（研磨会磨损），架构零改动。**
真要做耐久系统，请**单独立项**（它是"武器唯一实例"这个地基问题）。

⚠️ 顺带：**锋刃研磨的材料被用户清空了**（原本石料×2）。
如果它保持"永久 +75% 穿透且不要材料"，那是**免费的大幅增益**。疑似误删，**建议确认**。

## 9. 🔴 流血两条：引擎新轴 + 配平闸门要重算

### 现状：**武器完全不参与流血速率**

```
失血/秒 = Σ(每处伤口 × 分级权重) × BleedRatePerWound × BleedRateMultiplier
```
- **分级权重**（`BleedModel.RateWeightOf`）：致命 1.0 / 轻微 0.5 / 微小 0.2 —— **由受伤部位决定**
- **`BleedRatePerWound`**：`Body.cs:94` 默认 0.55，`Duel.cs:100` 用 **1.5**
- **`BleedRateMultiplier`**：**受害者**体质（丧尸 = **1/3**）

⇒ **`Weapon` 类没有任何流血字段。** 而且伤口目前只是**计数**（每部位最多 3 处）——
要让"这个伤口是锯齿剑划的、流得更快"成立，**伤口得从计数变成带属性的对象**。这是个不小的结构改动。

### ⚠️ 配平闸门警告（`BleedModel.cs:47-50` 原文）

> 丧尸 1/3 流血率同时是"流血加强"这条改动的**配平闸门**：**没有它，锐器只要划两刀站着等丧尸流干就行，
> 伤害与护甲都会失去意义。**

流血**刚被大幅加强**（用户要"多个严重伤口能在分出胜负前放干人"）。
**在这个基础上再给锐器 +40% 流血速度，很可能直接触发它自己警告的那个失效模式。**

**建议**：要做的话**必须先跑 Sim 复验**（锯齿剑刃 vs 丧尸 / vs 劫掠者，看流血致死占比），
且大概率要**同步下调 `BleedRatePerWound` 或提高丧尸抗性**来补偿。**别加了就走。**

「钉子强化：25% 几率造成小流血」同理，还额外需要"概率附加一个 Micro 伤口"的机制（走 `IRandomSource`）。

## 14. 总评

**用户对算符的直觉是对的，而且比"照抄 CLAUDE.md 铁律"更深一层**：
他知道百分比该乘算，也知道**棍棒的零穿透必须用加算救**——
这两件事看似矛盾，实则是同一条数学性质的两面（乘算在零上永远是零：
在"操作能力"那里我们**要**这个性质，在"穿透"这里我们**不要**）。
这条洞察值得写成通则 + 护栏，免得以后有人把加算"统一"成乘算。

**他的白名单收窄同时解决了三个问题**：叠加顺序分叉、穿透溢出、以及上一轮我提的"截短绕过冲锋枪不可双持"。
**约束不靠运算顺序去兜，靠装配规则直接杜绝** —— 这比定一个叠加顺序干净得多。

**枪托型态削得非常准**：三个型态（1.90 / 待定 / 2.29）全部落在锚点内，
**DPS 排序与重量代价排序完全单调一致**。

**材料合并是最该做的一条** —— 不是为了"简化"，而是因为**金属锭现在根本拿不到**，吃它的 9 样东西全是死内容。

**而用户最看重的「重量作为核心代价」，恰恰是唯一一条在当前代码里完全落不了地的** ——
装备不进负重账、debuff 没接线。**这不是把数字调大能解决的**，需要一个跨系统的独立批次。
**建议先把这件事的性质告诉用户，让他决定：是立项做，还是先接受"重量只是 flavor"。**
